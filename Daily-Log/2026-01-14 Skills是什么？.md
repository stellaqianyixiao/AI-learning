# 什么是Skills？我们能怎么使用Skills  

---

## 一、背景与问题意识

在 Agent 成为 AI 产品核心形态之后，一个关键问题逐渐凸显：  
**如何让通用大模型稳定、低成本、可复用地完成“专业级任务”？**

Anthropic 系统性提出并工程化落地了 **Skills（Agent Skills）** 这一概念，随后被 OpenAI（Codex）、Cursor 等生态快速采纳，并形成了跨平台的开放标准。Skills 并非又一个“插件”或“工具接口”，而是一种 **面向 Agent 的能力封装与分发范式**，其影响远大于表面形态。

我将从定义、使用方式、产品现状与未来演进四个层面，对 Skills 做一次系统梳理。

---

## 二、什么是 AI 领域的 Skills

### 2.1 核心定义（产品级表述）

**Skills 是一种将“完成某类任务所需的流程性知识与执行规范”进行模块化封装的 Agent 能力单元。**

在 Anthropic 的定义中：

- Skill 不是 prompt
- 不是模型能力
- 也不是单一工具调用

而是一个**可移植、可版本化、可按需加载的能力包**，用于告诉 AI Agent：

> 当面对某一类任务时，应该“如何做事”，  
> 包括步骤、标准、判断规则、产出结构，以及必要时的可执行资源。

### 2.2 工程形态

在工程层面，一个 Skill 通常表现为一个目录结构：

```

/my-skill
├─ SKILL.md        # 核心定义文件
├─ assets/         # 模板、示例、规范
├─ references/     # 背景资料、规则说明
└─ scripts/        # 可选：可执行脚本或工具封装

```

其中：

- **`SKILL.md` 是入口文件**
  - 包含 YAML metadata（name / description / triggers 等）
  - 以及 Markdown 形式的操作指令（procedural instructions）
- 其余内容为“按需加载资源”，并不会默认进入模型上下文

### 2.3 Skills 本质上解决了什么问题

从产品视角看，Skills 解决的是三个长期存在但未被很好抽象的问题：

1. **流程性知识无法复用**  
   组织里真正有价值的不是“知道什么”，而是“知道该怎么一步步做”。

2. **Prompt 工程不可规模化**  
   长 prompt 难维护、难版本控制、难在团队中共享。

3. **Agent 输出不可控、不稳定**  
   缺乏统一流程与产出规范，导致质量波动与返工成本高。

Skills 的出现，本质上是把 **“专家做事方式”产品化**。

## 三、Skills 的工作机制

### 3.1 渐进式披露（Progressive Disclosure）

Anthropic 在 Claude 中明确提出了 Skills 的核心运行机制：**渐进式披露**。

从上下文加载角度看，Skill 内容分为三层：

1. **Metadata（常驻）**  
   - 模型只需知道「有哪些 skills」以及「各自适合做什么」

2. **Instructions（触发时加载）**  
   - 当任务匹配某个 skill 时，加载其操作规范

3. **Resources / Scripts（必要时加载或执行）**  
   - 仅在确实需要时才访问模板、参考资料或脚本

这一机制带来两个直接结果：

- Skills 的信息容量不再受上下文窗口严格限制  
- Agent 可以在复杂任务中保持“认知聚焦”

### 3.2 Skills 与 Tools / MCP / 插件的区别

从产品架构上看，可以这样区分：

| 概念 | 解决的问题 |
|----|----|
| Tools | 能做什么原子操作 |
| MCP（Model Context Protocol） | 能连接哪些外部系统与数据 |
| Plugins / Extensions | 能力入口与分发 |
| **Skills** | **拿到能力之后，复杂任务应该如何被系统性完成** |

一个成熟的 Agent 产品，必然是 **MCP × Tools × Skills** 的组合，而不是三选一。

## 四、如何使用 Skills（现阶段产品形态）

### 4.1 Anthropic 生态（原生支持）

Anthropic 已在多个产品中原生支持 Skills：

- Claude.ai
- Claude Code
- Claude API
- Claude Agent SDK

当前可直接使用的官方 skills 包括：

- `docx`：Word 文档生成与编辑
- `xlsx`：Excel 创建与分析
- `pptx`：PPT 生成
- `pdf`：PDF 输出

这些并非简单的“文件导出”，而是完整的文档生成流程封装。

### 4.2 OpenAI 生态（以 Codex 为核心）

OpenAI 在 **Codex CLI 与 IDE Extensions** 中明确支持 Agent Skills，并采用与 Anthropic 一致的开放标准。

- Skills 可作为代码开发中的“可复用工作流模块”
- OpenAI 维护了官方 skills catalog 仓库
- 技术上强调：版本控制、团队共享、可组合

相比之下，ChatGPT 本身对 Skills 的支持仍处于逐步渗透阶段，更适合作为趋势判断而非稳定能力依赖。

### 4.3 第三方 Agent 产品

- **Cursor**：已在 Agent 能力中引入 Skills 概念，强调可移植与自动发现
- 部分 Agent IDE / Workflow 工具正在围绕 Skills 构建能力市场

## 五、Skills 的典型应用方式（产品案例）

### 5.1 将企业 SOP 转化为可执行能力

**示例：合规审查 Skill**

- 明确适用文档类型与审查范围
- 固化审查 checklist 与风险分级规则
- 输出结构稳定的审查报告（问题 + 引用 + 修改建议）

价值不在“模型更聪明”，而在于：
> 把资深专家的做事方式，变成组织级基础设施。

### 5.2 数据与增长分析的流程固化

**示例：增长周报 Skill**

- 指标口径定义
- 异常判断逻辑
- 结论输出结构（现象 → 原因 → 行动）

结果是：
- 口径一致
- 结论可比较
- 新人也能产出“看起来像专家”的分析

### 5.3 内容与品牌治理

**示例：品牌内容 Skill**

- 语气规范、禁用词
- 内容结构模板
- CTA 与多语言一致性规则

Skills 在这里承担的是“品牌治理工具”的角色，而非内容生成器。

## 六、Skills 将如何改变 AI 产品形态

### 6.1 从“助手”到“岗位化 Agent”

Skills 的长期影响在于：  
**AI 产品的核心资产将从“模型能力”转向“技能库”。**

未来的 Agent 不再被定义为“一个聪明的聊天对象”，而是：

- 一个拥有明确岗位边界的数字员工
- 能力由 Skills 组合决定，而非模型参数

### 6.2 能力边界的扩展方式发生变化

传统路径：  
> 新能力 = 新模型 / 新微调 / 新工程流程

Skills 路径：  
> 新能力 = 新 Skill（流程封装） + 现有模型 + 现有工具

这会显著降低能力扩展的边际成本。

### 6.3 效率提升来自“稳定性”，而非“智商提升”

Skills 带来的效率提升主要体现在：

- 更少 prompt 拼装
- 更少上下文冗余
- 更稳定的输出结构
- 更低的返工与评审成本

这是**工程效率与组织效率**的提升，而不仅是模型性能提升。

## 七、高潜产品应用场景（按组织价值密度）

1. 文档密集型工作（投研、法务、咨询、PMO）
2. 数据与增长运营
3. 软件研发全流程（PR Review、Release Notes、复盘）
4. 客服与质检
5. 销售方案与交付流程

这些场景的共同特征是：

- 流程明确
- 产出结构稳定
- 人工执行成本高

---

## 八、风险与治理问题

Skills 并非“纯内容”，而是可能包含可执行逻辑，因此必须正视：

- 供应链风险（来源不可信）
- 执行权限风险（脚本 / 文件 / 网络）
- 数据泄露风险

在企业级产品中，Skills 必须配套：

- 来源标记与签名机制
- 权限最小化
- 沙箱执行
- 完整审计日志
- 更新回归测试

否则，Skills 的规模化只会放大风险。

---

## 九、对 AI 产品路线图的启示

从产品规划角度，Skills 不应被当作“一个功能点”，而应被视为**平台级能力**：

1. 从高频 SOP 入手构建 Skill Library
2. 产品化 Skill 创建体验（模板 / 校验 / 生成）
3. 建立 Skill 运营指标（触发率、成功率、节省时间）
4. 支持组织级分发与版本治理
5. 与 MCP / Tool 生态深度结合
